# Character Fit Analysis — Review & Integration Assessment

## What You've Sent vs. What Exists

The existing system (from [`role_alignment_proposal.md`](file:///e:/Pro/audition-coach/Docs/role_alignment_proposal.md) and [`performance.ts`](file:///e:/Pro/audition-coach/apps/web/src/app/actions/performance.ts)) is a **4-parameter flat evaluation**:

| Existing (4 params) | New Proposal (8 dimensions) |
|---|---|
| Visual & Castability Fit | Archetype Embodiment |
| Physicality & Posture | Emotional Arc Fidelity |
| Vocal Characterization | Status Portrayal |
| Character Essence (Subtext) | Energy Signature Match |
| — | Psychological Core Visibility |
| — | Physical & Vocal Alignment *(merged)* |
| — | Overall Casting Fit *(synthesis)* |
| — | Screen Presence & Visual Character Fit *(sub-scored)* |

**The new proposal is a complete supersession of the old one.** It is fundamentally more sophisticated and director-facing.

---

## What's Excellent

### 1. The `CharacterBrief` Struct Is the Right Abstraction
The `buildCharacterFitBlock` function takes a typed `CharacterBrief` object and injects every field into the prompt at the right level of granularity. This is far more maintainable than building prompt strings with raw DB columns. It also naturally maps to an extended `script` table schema.

### 2. Dimension Design Is Psychologically Precise
The "Flag if" instructions in each dimension are **the real value** here. They teach the model what *not* to do — not just what to look for. This is the difference between a checklist scorer and an actual casting director's eye. Most AI evaluation prompts don't do this.

### 3. Screen Presence Sub-Scores Are Defensible
Breaking `screen_presence` into 4 sub-scores (`camera_expressiveness`, `presentation_register`, `physical_precision_match`, `visual_world_coherence`) with explicit exclusions (no beauty, no ethnicity, no body type) makes this **legally and ethically safe** to ship. The fallback to `null` with a message when video quality is insufficient is also the correct pattern.

### 4. `fit_vs_performance_gap` Is a Unique Signal
The gap between `character_fit.casting_fit_score` and `overall_score` (from the existing performance analysis) is genuinely useful information that doesn't exist today. It answers the casting director's real question: *"Can this actor perform well but still not be right for this role?"*

### 5. `casting_label` Thresholds Are Clear
Four labels (Not the right fit / Interesting but misaligned / Strong consideration / Call them back) map cleanly to how real casting works and are immediately UI-renderable without logic in the frontend.

## How It Fits Into the Current System

### Gemini Call Architecture — Two Calls, One Upload

> [!IMPORTANT]
> **Decision: Two separate Gemini calls sharing a single video upload.** This eliminates JSON truncation risk entirely. Each call has a focused, bounded output. If character fit fails, the existing line analysis is already saved and unaffected.

```
Upload video once → fileUri
      │
      ├── Call 1 (unchanged): Existing prompt Steps 1–6
      │     → Returns: { overall_score, metrics_avg, line_analysis, errors, directors_notes }
      │     → Saved to: performance_analysis, analysis_lines, analysis_errors
      │
      └── Call 2 (new, conditional): buildCharacterFitBlock(brief)
            → Only runs if script has brief fields populated in DB
            → Returns: { character_fit: { casting_fit_score, casting_label, dimensions, ... } }
            → Saved to: character_fit_analysis
      │
      └── Delete file once, after both calls complete
```

**Call 2 implementation sketch in [`performance.ts`](file:///e:/Pro/audition-coach/apps/web/src/app/actions/performance.ts):**

```typescript
// ─── Call 1: Existing (unchanged) ────────────────────────────────
const perfResult = await model.generateContent([
  existingPrompt,
  { fileData: { fileUri: uploadResult.file.uri, mimeType: uploadResult.file.mimeType } }
]);
const analysis = JSON.parse(perfResult.response.text());
// ...save to performance_analysis, analysis_lines, analysis_errors (unchanged)

// ─── Call 2: Character Fit (conditional) ─────────────────────────
const brief = buildCharacterBriefFromScript(performance.script); // maps DB columns → CharacterBrief
if (brief) {
  const fitModel = genAI.getGenerativeModel({
    model: "gemini-2.5-flash",
    generationConfig: {
      responseMimeType: "application/json",
      maxOutputTokens: 8192,   // ← explicit ceiling
    }
  });

  const fitResult = await fitModel.generateContent([
    buildCharacterFitBlock(brief),
    { fileData: { fileUri: uploadResult.file.uri, mimeType: uploadResult.file.mimeType } }
    //           ↑ same URI — no re-upload
  ]);

  let characterFit;
  try {
    characterFit = JSON.parse(fitResult.response.text());
  } catch (parseError) {
    // Truncation recovery — trim to last valid closing brace
    const raw = fitResult.response.text();
    const trimmed = raw.substring(0, raw.lastIndexOf('}') + 1);
    characterFit = JSON.parse(trimmed);
  }
  // ...save characterFit to character_fit_analysis table
}

// ─── Cleanup: single delete after both calls ──────────────────────
await fileManager.deleteFile(uploadResult.file.name).catch(console.error);
```

| | Single combined call | ✅ Two calls, one upload |
|---|---|---|
| Truncation risk | High — one long combined output | **None** — each output is focused |
| Existing analysis safety | Fails together with character fit | **Existing saves first, independently** |
| Re-upload cost | None | **None — same fileUri reused** |
| Extra latency | — | ~1–2s |
| Maintainability | Coupled prompts | **Fully independent prompts** |

## `CharacterBrief` Type — Shared Contract

**New file: [`src/lib/types/character-brief.ts`](file:///e:/Pro/audition-coach/apps/web/src/lib/types/character-brief.ts)**

```typescript
// src/lib/types/character-brief.ts

export type VisualBrief = {
  screen_presence_note: string;
  physical_world_fit: string;
  role_specific_physicality: string;
};

export type CharacterBrief = {
  archetype: string;
  emotional_arc: string;
  status_in_scene: string;
  energy_signature: string;
  psychological_core: string;
  physical_presence: string;
  voice_character: string;
  director_vision_note: string;
  visual_brief: VisualBrief;
};
```

This type is the **single source of truth** consumed by:
- `buildCharacterFitBlock(brief: CharacterBrief)` — the prompt builder
- `buildCharacterBriefFromScript(script)` — the DB-to-type mapper in `performance.ts`
- Any future UI component that reads or displays the brief

**Mapper function in `performance.ts`** (converts DB row → `CharacterBrief | null`):

```typescript
import type { CharacterBrief } from '@/lib/types/character-brief';

function buildCharacterBriefFromScript(script: any): CharacterBrief | null {
  // Returns null if the script has no brief defined — skips Call 2 entirely
  if (!script.archetype) return null;

  return {
    archetype:            script.archetype,
    emotional_arc:        script.emotional_arc,
    status_in_scene:      script.status_in_scene,
    energy_signature:     script.energy_signature,
    psychological_core:   script.psychological_core,
    physical_presence:    script.physical_presence,
    voice_character:      script.voice_character,
    director_vision_note: script.director_vision_note,
    visual_brief: {
      screen_presence_note:      script.screen_presence_note,
      physical_world_fit:        script.physical_world_fit,
      role_specific_physicality: script.role_specific_physicality,
    }
  };
}
```

TypeScript will now catch mismatches at write time — wrong field names, missing fields, wrong nesting — across every file that imports this type.

## What Needs to Change

### `script` table — new nullable columns
```sql
ALTER TABLE script
  ADD COLUMN archetype                 TEXT,
  ADD COLUMN emotional_arc             TEXT,
  ADD COLUMN status_in_scene           TEXT,
  ADD COLUMN energy_signature          TEXT,
  ADD COLUMN psychological_core        TEXT,
  ADD COLUMN physical_presence         TEXT,
  ADD COLUMN voice_character           TEXT,
  ADD COLUMN director_vision_note      TEXT,
  ADD COLUMN screen_presence_note      TEXT,
  ADD COLUMN physical_world_fit        TEXT,
  ADD COLUMN role_specific_physicality TEXT;
```
All nullable — existing scripts have `NULL` for all fields, which causes `buildCharacterBriefFromScript` to return `null` and skip Call 2 entirely. Zero impact on existing rows.

### New `character_fit_analysis` table
```sql
CREATE TABLE character_fit_analysis (
  id                      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  performance_id          UUID REFERENCES performance(id) ON DELETE CASCADE,
  casting_fit_score       INTEGER CHECK (casting_fit_score BETWEEN 0 AND 100),
  casting_label           TEXT,
  -- dimension scores
  archetype_score         INTEGER,
  archetype_comment       TEXT,
  emotional_arc_score     INTEGER,
  emotional_arc_comment   TEXT,
  status_score            INTEGER,
  status_comment          TEXT,
  energy_score            INTEGER,
  energy_comment          TEXT,
  psych_core_score        INTEGER,
  psych_core_comment      TEXT,
  phys_vocal_score        INTEGER,
  phys_vocal_comment      TEXT,
  screen_presence_score   INTEGER,    -- nullable
  screen_presence_comment TEXT,
  -- screen presence sub-scores (all nullable)
  camera_expressiveness   INTEGER,
  presentation_register   INTEGER,
  physical_precision      INTEGER,
  visual_world_coherence  INTEGER,
  -- synthesis fields
  fit_vs_performance_gap  TEXT,
  director_recommendation TEXT,
  created_at              TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

## Open Engineering Question

> [!NOTE]
> **Overall casting fit score (dimension 7) is a weighted synthesis — but the weights aren't defined.** You should decide whether the synthesis is: (a) a simple average of all 7 dimension scores in post-processing, (b) a prompt-level instruction to the model to weight qualitatively, or (c) a hardcoded weighted formula (e.g., archetype × 0.2 + emotional arc × 0.15...). Option (b) is what the current prompt does implicitly — which is fine, but scores may vary slightly across runs.
